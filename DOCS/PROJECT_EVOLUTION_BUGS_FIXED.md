##Цей документ фіксує всі критичні вразливості, архітектурні "діри" та баги, які існували в проєкті на момент початку нашої роботи, і які були успішно виправлені командою FAE. Це наочна демонстрація того, чого ми досягли спільними зусиллями.

## 1. Архітектурні та Концептуальні "Діри"
* **Анемічна модель предметної області (Anemic Domain Model):** Сутності були "мертвими" структурами даних без власної поведінки. Вся бізнес-логіка була розмита по хендлерах та роутерах. **Виправлено:** Впроваджено Rich Domain Model.
* **Відсутність транзакційності (No Unit of Work):** Збереження в базу відбувалося напряму в роутерах. При збоях база залишалася в неконсистентному стані. **Виправлено:** Додано патерн Unit of Work.
* **Баг Каскадного Збереження:** Синхронізація аплікаторів та машин працювала некоректно через відсутність атомарних транзакцій. **Виправлено:** Усунуто завдяки UoW.
* **Витік інфраструктури (Leaky Hexagonal):** Презентаційний шар (FastAPI) напряму звертався до сирих запитів бази даних. **Виправлено:** Впроваджено IoC-контейнер (`dishka`) та ізольовані адаптери.

## 2. Безпека та Авторизація (Критичні Вразливості)
* **Hardcoded Секрети:** Паролі бази даних (`docker-compose.yml`) та ключ `SECRET_KEY` (для JWT) лежали у відкритому вигляді прямо в коді. **Виправлено:** Винесено в `.env` та захищено через `pydantic-settings`.
* **Зламаний доступ (Broken Access Control):** Ендпоінти дашборду та веб-сокетів не вимагали токена, будь-хто міг переглядати статистику. **Виправлено:** Додано JWT-аутентифікацію.
* **Підвищення привілеїв (Privilege Escalation):** Звичайні Оператори могли сканувати вигадані номери і система автоматично створювала для них нові аплікатори в базі (в обхід прав Технічних Адмінів). **Виправлено:** Закрито, тепер видає `404 Not Found`.
* **Вразливий CORS:** Дозволялися запити з будь-якого домену (`allow_origins=["*"]`). **Виправлено:** Налаштовано "білий список" доменів через змінні оточення.
* **XSS вразливість на Frontend:** Використовувався небезпечний `innerHTML` для рендеру. **Виправлено:** Замінено на безпечний `textContent`.
* **Docker Root User:** Контейнери запускалися від імені суперкористувача. **Виправлено:** Створено multi-stage збірку з непривілейованим користувачем.

## 3. Конкурентність та Продуктивність (Високе Навантаження)
* **Стани гонитви (Race Conditions):** Ліміти місткості перевірялися без блокування бази. Одночасне сканування двох аплікаторів могло призвести до переповнення машини вище дозволеного ліміту. **Виправлено:** Додано Оптимістичне блокування (Optimistic Concurrency).
* **Необроблений StaleDataError:** При конфлікті транзакцій система "падала" з HTTP 500. **Виправлено:** Додано глобальний обробник, що переводить такі конфлікти у м'який статус `HTTP 409 Conflict`.
* **Thundering Herd у WebSockets:** Сервер кожні 2 секунди опитував всю базу даних (`asyncio.sleep(2)`) для оновлення дашборду. При сотнях клієнтів база б "лягла". **Виправлено:** Впроваджено подієву PubSub модель через `broadcaster` на базі PostgreSQL (оновлення "пушиться" миттєво лише при зміні).
* **--reload у Продакшені:** Сервер запускався з dev-прапорцем `--reload`. **Виправлено:** Запущено через промисловий сервер `gunicorn` з 4-ма воркерами.
* **Некоректний проксі Nginx:** Nginx не вмів обробляти WebSockets (були відсутні `Upgrade` заголовки). **Виправлено:** Конфігурацію оновлено.

## Висновок
Всі вищезазначені "діри", які тягнулися від стадії прототипу, було успішно ідентифіковано, протестовано і ліквідовано.
